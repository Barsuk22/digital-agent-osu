# План разработки автономного PPO-агента для Minecraft

## 0. Главная цель проекта

Мы хотим сделать не просто Minecraft-бота, который выполняет команды.

Мы хотим сделать автономную систему, которая:

- видит мир;
- понимает своё состояние;
- сама выбирает цели;
- выживает без помощи человека;
- добывает ресурсы;
- крафтит;
- строит;
- исследует;
- защищается;
- учится на задачах;
- умеет играть в одиночке;
- умеет играть на локальном сервере;
- потом умеет играть с друзьями;
- позже получает LLM-командира сверху;
- в финале становится частью общей системы агентов.

Главный принцип:

**Не делать одного огромного тупого PPO-монстра на весь Minecraft.**

Правильный путь:

```text
маленькие навыки -> общий менеджер навыков -> автономный агент -> серверный агент -> LLM-командир
```

То есть сначала делаем "ручки, ножки, глазки", потом маленькие умения, потом выживание, потом самостоятельное мышление.

## 1. Общая архитектура системы

Финальная система должна быть построена слоями.

Важно: агент не должен быть одним огромным PPO-монолитом.

Нужна иерархия:

```text
Personality / LLM Layer          # позже
        ↓
Intent / Goal Manager            # что сейчас имеет смысл делать
        ↓
Task Planner                     # как разложить цель на шаги
        ↓
Skill Manager                    # какой навык запустить
        ↓
Skills Layer                     # конкретные навыки
        ↓
Action Controller                # клавиши, мышь, пакеты, API
        ↓
Minecraft
```

### 1.1 Главная идея управления

LLM в будущем не должна управлять каждым кадром.

Она не должна напрямую нажимать:

```text
W, A, S, D, Space, LMB, RMB
```

Она должна работать на уровне намерений и решений:

```text
помочь игроку
добыть ресурс
построить укрытие
вернуться домой
исследовать место
поддержать игрока в бою
отказаться от плохой идеи
```

Нижние слои уже сами решают, как физически выполнить действие в Minecraft.

Правильное разделение:

```text
LLM / Personality Layer -> смысл, характер, общение, высокоуровневые решения
Goal Manager            -> приоритеты, выживание, выбор текущей цели
Task Planner            -> разбиение цели на шаги
Skill Manager           -> запуск и контроль навыков
Skills                  -> конкретное выполнение
Action Controller       -> низкоуровневые действия
```

### 1.2 Важное правило про личность

В будущем это не должен быть бездушный ассистент-командоисполнитель.

Это цифровой персонаж с характером.

Поэтому верхний слой должен воспринимать сообщения игроков не как абсолютные приказы, а как социальные входы:

```text
игрок попросил
игрок предложил
игрок позвал
игрок предупредил
игрок пошутил
игрок требует что-то опасное
```

Решение всё равно проходит через:

```text
личность -> контекст -> безопасность -> цели -> возможности -> действие
```

То есть даже если игрок говорит что-то вроде "иди туда", персонаж в будущем должен оценивать:

- безопасно ли это;
- есть ли сейчас более важная проблема;
- хочет ли она это делать;
- поняла ли она просьбу;
- не сломает ли это базу;
- не помешает ли другим игрокам;
- есть ли ресурсы и навыки для выполнения.

Это не значит, что она должна спорить всегда.

Это значит, что она не должна быть тупой кнопкой "выполнить команду".

### 1.3 Режимы работы Minecraft

Система должна поддерживать все режимы, а не выбирать один:

1. тренировочная среда;
2. одиночная игра;
3. локальный сервер;
4. личный сервер с друзьями;
5. позже - более сложные серверные сценарии.

Каждый режим нужен:

```text
training env      -> быстро учить и тестировать навыки
singleplayer      -> проверять настоящее survival-поведение
local server      -> проверять серверную синхронизацию
friends server    -> проверять социальное и командное поведение
```

### 1.4 Observation Builder

Это слой, который собирает всё, что агент "видит" и "чувствует".

Он должен собирать:

- картинку экрана;
- здоровье;
- голод;
- броню;
- воздух под водой;
- координаты;
- направление взгляда;
- текущий биом;
- время суток;
- выбранный слот;
- предмет в руке;
- инвентарь;
- ближайшие блоки;
- ближайших мобов;
- ближайших игроков;
- события: урон, смерть, подбор предмета, сломанный блок, созданный предмет.

### 1.5 Vision Encoder

Агент должен иметь зрение.

Базовый вариант:

```text
screen -> 84x84 или 128x128 -> CNN -> visual features
```

Лучше давать не один кадр, а несколько последних:

```text
4-8 последних кадров
```

Minecraft не требует реакции как osu, поэтому можно начинать с частоты:

```text
5-10 наблюдений в секунду
```

То есть примерно каждые:

```text
100-200 ms
```

### 1.6 State Encoder

Кроме картинки, агенту нужны числовые данные:

```text
hp, hunger, armor, selected_slot, inventory_summary, position, yaw, pitch
```

Это нельзя заставлять модель угадывать только глазами.

Правильно:

```text
картинка + состояние + память
```

### 1.7 Memory Module

Minecraft требует памяти.

Агент должен помнить:

- где был дом;
- где видел дерево;
- где была шахта;
- где лежали предметы;
- что он уже пытался сделать;
- почему умер;
- какая цель сейчас активна;
- какие ресурсы нужны.

Для первой версии достаточно:

```text
GRU или LSTM внутри политики
```

Позже можно добавить внешнюю память:

```text
world memory / map memory / event memory / social memory
```

### 1.8 Skills Layer

Навык - это не обязательно PPO-агент.

Навыки делятся на три типа:

```text
ScriptSkill      # строгое действие по правилам
HeuristicSkill   # правила + проверки + простая логика
PPOSkill         # обучаемое поведение
```

Примеры:

```text
CraftItem        -> ScriptSkill / HeuristicSkill
InventoryMove    -> ScriptSkill
UseChest         -> ScriptSkill / HeuristicSkill
MoveToTarget     -> HeuristicSkill + иногда PPO
MineBlock        -> HeuristicSkill + PPO
Combat           -> PPOSkill
EscapeDanger     -> PPOSkill / HeuristicSkill
FollowPlayer     -> HeuristicSkill + PPO
BuildBlueprint   -> ScriptSkill + HeuristicSkill
```

Главное правило:

```text
Не обучать через PPO то, что можно надёжно сделать правилами.
```

PPO нужен там, где есть:

- движение в сложном мире;
- непредсказуемость;
- бой;
- уклонение;
- визуальное управление;
- адаптация;
- частые ошибки из-за окружения.

### 1.9 Skill Manager

Skill Manager - диспетчер навыков.

Он должен:

- запускать навык;
- передавать цель навыку;
- следить за прогрессом;
- останавливать навык;
- понимать успех;
- понимать провал;
- делать retry;
- переключаться на другой навык;
- прерывать действие при опасности.

Пример:

```text
Goal: get_wood
Task Planner:
  1. find_tree
  2. move_to_tree
  3. mine_log
  4. collect_drop
  5. repeat_until wood >= target_count

Skill Manager:
  run FindTree
  run MoveToTarget(tree)
  run MineBlock(log)
  run CollectItem(log_drop)
```

### 1.10 Goal Manager

Goal Manager решает, что сейчас важнее.

Он должен учитывать:

- здоровье;
- голод;
- угрозы;
- ночь/день;
- текущие ресурсы;
- текущую цель;
- просьбы игроков;
- память мира;
- риск смерти;
- возможности агента.

Пример приоритетов:

```text
if hp_low:
    goal = survive
elif hostile_near and weak:
    goal = escape
elif hunger_low:
    goal = get_food
elif night and no_shelter:
    goal = build_or_find_shelter
elif external_request_active and safe:
    goal = handle_request
else:
    goal = improve_survival_state
```

### 1.11 Task Planner

Task Planner превращает цель в цепочку задач.

Пример:

```text
Goal: craft_stone_pickaxe
Plan:
  1. ensure wood >= 3
  2. ensure crafting_table exists
  3. craft planks
  4. craft sticks
  5. craft wooden_pickaxe if needed
  6. mine stone >= 3
  7. craft stone_pickaxe
```

Task Planner не нажимает кнопки.

Он только строит план.

### 1.12 Action Controller

Action Controller - самый нижний слой.

Он знает, как физически выполнить действие:

- нажать клавишу;
- отпустить клавишу;
- повернуть камеру;
- кликнуть мышью;
- выбрать слот;
- использовать item;
- отправить команду через API;
- взаимодействовать через Mineflayer.

Верхние слои не должны знать детали клавиш.

### 1.13 Autonomy Loop

Финальная автономия выглядит так:

```text
1. Получить наблюдение
2. Обновить память
3. Проверить угрозы
4. Проверить внутреннее состояние
5. Проверить внешние сообщения/просьбы
6. Выбрать цель
7. Построить план
8. Запустить навыки
9. Следить за результатом
10. Прервать при опасности
11. Записать событие в память
12. Выбрать следующую цель
```

## 4. Фаза C - зрение и состояние

### Цель

Сделать нормальные "глазки" агента.

### Входы модели

Минимальный вход:

```text
screen_84x84
last_4_frames
hp
hunger
armor
selected_slot
inventory_summary
yaw/pitch

```

Расширенный вход:

```text
nearby_blocks

nearest_player

```

### Почему не только скриншоты

Потому что если дать только картинку, агенту придётся угадывать слишком много.

Он может увидеть дерево, но не знать:

- сколько у него дерева уже есть;
- голодный ли он;
- скоро ли ночь;
- есть ли кирка;
- где дом.

Поэтому правильно:

```text
картинка + состояние + память
```

### Что должно получиться

К концу фазы есть стабильный поток наблюдений, который можно давать модели.

## 5. Фаза D - тренировочная среда

### Цель

Сделать безопасную песочницу, где агент будет учиться маленьким действиям.

### Почему нужна тренировочная среда

Нельзя сразу бросать агента в обычный survival.

Он будет:

- смотреть в землю;
- бить воздух;
- умирать;
- теряться;
- ломать не то;
- стоять как печальная картошка.

Поэтому сначала нужны маленькие учебные миры.

### Учебные сценарии

#### D1. Смотреть на цель

Цель:

```text
повернуть камеру на нужный блок
```

Примеры:

- посмотреть на дерево;
- посмотреть на землю;
- посмотреть на верстак;
- посмотреть на моба.

#### D2. Дойти до блока

Цель:

```text
подойти к нужному блоку на дистанцию взаимодействия
```

Примеры:

- подойти к дереву;
- подойти к камню;
- подойти к воде;
- подойти к сундуку.

#### D3. Ломать блок

Цель:

```text
смотреть на блок -> удерживать действие -> получить предмет
```

Блоки:

- dirt;
- wood;
- leaves;
- stone;
- coal ore;
- iron ore.

#### D4. Подбирать предмет

Цель:

```text
увидеть выпавший предмет -> подойти -> подобрать
```

#### D5. Ставить блок

Цель:

```text
выбрать блок -> выбрать место -> поставить
```

#### D6. Крафтить

Цель:

```text
иметь ресурсы -> открыть интерфейс -> создать предмет
```

Предметы:

- planks;
- sticks;
- crafting table;
- wooden pickaxe;
- stone pickaxe;
- furnace;
- torch.

#### D7. Есть еду

Цель:

```text
понять голод -> выбрать еду -> съесть
```

#### D8. Бой

Цель:

```text
заметить угрозу -> повернуться -> атаковать / убежать
```

Мобы:

- zombie;
- spider;
- skeleton;
- creeper.

#### D9. Убегать от опасности

Цель:

```text
если угроза сильнее -> отступить -> найти безопасное место
```

#### D10. Вернуться домой

Цель:

```text
помнить точку дома -> вернуться
```

### Что должно получиться

К концу фазы есть набор маленьких задач, на которых можно тренировать навыки.

## 6. Фаза E - imitation learning

### Цель

Сначала показать агенту человеческие примеры, чтобы PPO не учился с полного нуля.

### Что записываем

Нужно записать демонстрации:

- как рубить дерево;
- как собирать дерево;
- как делать доски;
- как делать верстак;
- как делать палки;
- как делать кирку;
- как копать камень;
- как строить укрытие;
- как есть;
- как убегать от моба;
- как возвращаться домой;
- как идти за игроком.

### Формат данных

Каждый шаг:

```text
observation
pressed_keys
mouse_delta
action
inventory_state
reward_hint
done
```

### Что обучаем

Первую модель поведения:

```text
наблюдение -> действие как у человека
```

Это не финальный агент.

Это "малыш посмотрел, как взрослый делает".

### Что должно получиться

- дерево надо бить;
- предметы надо подбирать;
- ночью опасно;
- еду надо есть;
- верстак полезный.

## 7. Фаза F - PPO-навыки

### Цель

#### Skill 1 - LookAtTarget

Умеет смотреть на объект.

#### Skill 2 - MoveToTarget

Умеет идти к блоку, предмету, игроку или мобу.

#### Skill 3 - MineBlock

Умеет добывать блок.

#### Skill 4 - CollectItem

Умеет подобрать предмет.

#### Skill 5 - PlaceBlock

Умеет поставить блок.

#### Skill 6 - CraftItem

Умеет создать предмет.

#### Skill 7 - EatFood

Умеет восстановить голод.

#### Skill 8 - EscapeDanger

Умеет убегать от опасности.

#### Skill 9 - BasicCombat

Умеет драться с простыми мобами.

#### Skill 10 - FollowPlayer

Умеет идти за человеком.

### Reward для навыков

#### Пример MineBlock

```text
+ посмотрел на нужный блок
+ подошёл на нужную дистанцию
+ начал ломать правильный блок
+ блок сломан
+ предмет появился
+ предмет подобран
- бил воздух
- бил неправильный блок
- отошёл далеко
- получил урон
- умер
```

#### Пример FollowPlayer

```text
+ держит дистанцию 2-5 блоков
+ смотрит примерно на игрока
+ не застревает
+ обходит препятствия
- потерял игрока
- ушёл слишком далеко
- толкает игрока
- упал
```

### Что должно получиться

К концу фазы есть набор рабочих навыков.

Они пока не "думают глобально", но каждый делает свою маленькую работу.

## 8. Фаза G - менеджер навыков

### Цель

Собрать маленькие навыки в цепочки действий.

### Примеры цепочек

#### Получить дерево

```text
Find tree
MoveToTarget(tree)
LookAtTarget(log)
MineBlock(log)
CollectItem(log)
```

#### Сделать верстак

```text
Check inventory: logs
CraftItem(planks)
CraftItem(crafting_table)
PlaceBlock(crafting_table)
```

#### Сделать деревянную кирку

```text
Get logs
Craft planks
Craft sticks
Craft crafting table
Place crafting table
Craft wooden pickaxe
```

#### Пережить ночь

```text
Check time
Find safe place
Build shelter
Place torch if available
Stay inside
Fight/escape if mob enters
Eat if hungry
```

### Ошибки, которые менеджер должен понимать

- навык не завершился;
- цель потеряна;
- предмета нет;
- инструмент сломался;
- путь заблокирован;
- моб атакует;

## 9. Фаза H - survival loop

### Цель

инструменты

### Базовый survival loop

```text
1. Проверить здоровье

8. Если голод - искать/есть еду
9. Если ночь - укрытие
10. Если безопасно - выполнять долгую цель
11. Обновить память
12. Повторить
```

### Цели survival-агента

Он должен уметь сам:

- сделать факелы;
- найти еду;

- пережить ночь;
- исследовать рядом;
- не терять дом;
- возвращаться домой;
- складывать ресурсы в сундук;
- улучшать инструменты;
- избегать смерти.

### Что должно получиться

К концу фазы агент уже может жить в одиночке хотя бы простую раннюю игру.

Не идеально.

Но уже не беспомощный хлебушек.

## 12. Фаза K - локальный сервер

### Цель

Запустить агента не в одиночке, а на своём локальном сервере.

Это нужно, потому что серверный Minecraft отличается от одиночки.

### Что меняется

На сервере:

- другая задержка;
- мир может идти без паузы;
- события приходят иначе;
- могут быть другие игроки;
- возможны рассинхроны;
- команды выполняются не всегда идеально.

### Что агент должен уметь на локальном сервере

- подключиться;
- заспавниться;
- понять своё состояние;
- двигаться без рассинхрона;
- добывать блоки;
- подбирать предметы;
- пользоваться сундуками;
- следовать за игроком;
- не мешать игроку;
- возвращаться к базе;
- реагировать на чатовые команды-шаблоны.

### Примеры простых команд

```text
follow me
stop
come here
get wood
go home
drop items
protect me
mine stone
```

Пока без LLM.

Это просто команды-шаблоны.

### Что должно получиться

Агент умеет жить на локальном сервере и выполнять базовые команды.

## 17. Фаза P - LLM Commander

### Цель

Добавить LLM сверху, но не вместо агента.

LLM не должна нажимать кнопки напрямую.

LLM должна быть командиром.

### Правильная схема

```text
Player says: "Построй маленький дом рядом со мной"
    ↓
LLM understands goal
    ↓
LLM creates plan
    ↓
Goal Manager checks if possible
    ↓
Skill Manager executes skills
    ↓
Agent reports result
```

### Что LLM должна делать

- понимать просьбы;
- разбивать цель на шаги;
- выбирать приоритет;
- объяснять, что делает;
- спрашивать уточнение только если нужно;
- помнить стиль общения;
- общаться с игроками;
- давать агенту высокоуровневые цели.

### Что LLM не должна делать

- напрямую управлять каждым кадром;
- заменять PPO;
- принимать миллисекундные решения;
- ломать автономию;
- делать опасные действия без проверки.

### Что должно получиться

Агент становится понятным для людей.

С ним можно говорить.

Но тело и навыки уже работают сами.

## 18. Фаза Q - онлайн-обучение

### Цель

Добавить аккуратное дообучение после основного агента.

Это делается в самом конце.

И это правильно.

### Почему не раньше

Потому что онлайн-обучение на сыром агенте может испортить модель.

Он будет учиться на своих же глупостях.

Сначала нужен стабильный агент.

Потом уже:

```text
игра -> сбор опыта -> оценка -> фильтр -> дообучение -> тест -> обновление модели
```

### Как должно работать

1. агент играет;
2. система пишет опыт;
3. после сессии считает метрики;
4. плохие куски фильтруются;
5. хорошие куски идут в replay buffer;
6. новая модель обучается отдельно;
7. проходит тесты;
8. только потом заменяет старую.

### Важно

Нельзя просто обновлять боевую модель прямо во время игры без проверки.

Иначе она может внезапно стать картошкой и забыть, как ходить.

### Что должно получиться

Агент может постепенно улучшаться после игровых сессий.
